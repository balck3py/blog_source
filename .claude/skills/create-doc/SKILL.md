# 技术文档生成器

根据本知识库的 `rule.md` 规范生成标准化技术文档。

## 使用流程

### 1. 确定文档信息

先根据当前沟通的上下文自己拟定一个文档主题并发给用户确认，如果用户确认没问题就继续创建，否则根据用户的输入重新生成主题  
根据确定的主题和上下文，先初步判定文档应该属于哪个文件夹下，并给用户确认
向用户询问以下信息（如未提供）：
- 主要内容要点

### 2. 确定文件名和序号

检查目标目录中已有的文件，确定下一个序号：

```bash
ls -la "[目录路径]"
```

文件命名格式：`[序号]. [标题].md`（序号使用两位数字：01, 02, 03...）

### 3. 生成文档结构

#### 必需的 YAML Frontmatter

```yaml
---
title: [文档标题]
date: [当前日期 YYYY-MM-DD]
author: Eugen
tags:
  - [关键词1]
  - [关键词2]
  - [关键词3]
  # 5-10个核心关键词，按重要性排序
---
```

#### 文档结构模板

```markdown
# [emoji] [文档主标题]

> [简介：说明文档面向的读者和主要内容]

---

## 📚 目录

1. [章节1](#1-章节1)
2. [章节2](#2-章节2)
...

---

## 1. 章节标题

### 子章节标题（可带 emoji）

[内容...]

```[语言]
// 代码示例（必须有中文注释）
```

### 📝 小结

> [章节核心要点总结]

---

## 📚 总结

### 🎯 关键概念速查表

| 概念 | 说明 |
|------|------|
| **概念1** | 说明 |

### ✨ 最佳实践

1. ✅ **推荐做法**
2. ❌ **避免做法**

---

> 📖 本文档持续更新，如有问题欢迎讨论！
```

## 写作规范

### 语言风格
- 使用中文简体
- 技术术语首次出现时中英文对照：`垃圾回收 (Garbage Collection, GC)`
- 使用"你"而非"您"，保持友好专业的语气

### Emoji 使用

| Emoji | 用途 |
|-------|------|
| 🧹 🧵 🔐 | 文档主题图标 |
| 🤔 | 提出问题 |
| ✨ | 解决方案/特性 |
| 📚 | 目录/索引 |
| 📊 | 图表/数据 |
| 💡 | 提示/建议 |
| ⚠️ | 警告/注意 |
| ✅ | 正确做法 |
| ❌ | 错误做法 |
| 🔄 | 流程/循环 |
| 📝 | 小结/总结 |
| 🎯 | 核心要点 |
| 🚀 | 性能优势 |
| 🐢 | 性能问题 |

### 代码示例规范
- 必须指定语言类型（`csharp`、`bash`、`yaml`、`json` 等）
- 必须包含中文注释解释代码
- 使用 ✅/❌ 标记展示正确与错误做法对比

```markdown
// ❌ 不好：说明为什么不好
[错误代码示例]

// ✅ 好：说明为什么好
[正确代码示例]
```

### 图片引用
- 图片存放于 `00.assets/` 目录
- 使用 Obsidian 语法：`![[../00.assets/Pasted image YYYYMMDDHHMMSS.png]]`

### 特殊格式

警告提示：
```markdown
> ⚠️ **警告**：[警告内容]
```

重要提示：
```markdown
> 💡 **提示**：[提示内容]
```

## 质量检查清单

生成文档后确保：
- [ ] Frontmatter 格式正确，包含 title、date、author、tags
- [ ] 标题层级清晰，使用 emoji 增强可读性
- [ ] 代码示例完整且有中文注释
- [ ] 术语首次出现有中英文对照
- [ ] 每个主要章节有 `### 📝 小结`
- [ ] 长文档有 `## 📚 总结`
